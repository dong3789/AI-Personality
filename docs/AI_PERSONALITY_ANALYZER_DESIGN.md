# AI Personality Analyzer - 설계 문서

## 1. AI 모델 타입 정의

### 1.1 GPT-4 타입 (The Perfectionist)
**특징**
- 완벽주의적 접근
- 철저한 문서화와 주석
- 높은 코드 품질과 일관성
- 체계적인 프로젝트 구조

**판별 기준**
- README.md가 상세하고 구조화되어 있음
- 코드 주석 비율이 높음 (15% 이상)
- 일관된 커밋 메시지 패턴
- 테스트 코드 존재
- 이슈/PR에 상세한 설명

### 1.2 GPT-3.5 타입 (The Pragmatist)
**특징**
- 실용적이고 빠른 구현
- 충분한 문서화 (과하지 않음)
- 효율성 중시
- 범용적인 솔루션 선호

**판별 기준**
- README.md가 명확하지만 간결함
- 코드 주석이 적절함 (5-10%)
- 규칙적인 커밋 패턴
- 주류 언어/프레임워크 사용
- 빠른 이슈 해결

### 1.3 Claude Opus 타입 (The Architect)
**특징**
- 체계적이고 구조화된 설계
- 확장성과 유지보수성 중시
- 깊이 있는 분석과 계획
- 안전성과 에러 핸들링 강조

**판별 기준**
- 명확한 디렉토리 구조
- 설계 문서 존재
- 에러 핸들링 코드 비율 높음
- 타입 시스템 활용 (TypeScript, Rust 등)
- 신중한 커밋 (큰 변경 전 계획)

### 1.4 Claude Sonnet 타입 (The Balanced Creator)
**특징**
- 균형잡힌 접근
- 창의적이면서도 실용적
- 코드와 문서의 조화
- 협업 친화적

**판별 기준**
- 중간 수준의 문서화
- 다양한 기술 스택 활용
- 협업 활동 활발 (이슈, PR 리뷰)
- 코드 리뷰 참여도 높음
- 꾸준한 업데이트

### 1.5 Gemini 타입 (The Multi-Tasker)
**특징**
- 다재다능함
- 여러 프로젝트 동시 진행
- 다양한 언어와 기술 사용
- 실험적 시도

**판별 기준**
- 5개 이상의 프로그래밍 언어 사용
- 다양한 도메인의 프로젝트
- 많은 브랜치와 실험적 커밋
- 새로운 기술 빠른 도입
- 활발한 fork와 contribution

### 1.6 Llama 타입 (The Open Source Champion)
**특징**
- 오픈소스 정신
- 커뮤니티 중심
- 협업과 공유 강조
- 접근성과 사용성 중시

**판별 기준**
- 명확한 라이센스 명시
- CONTRIBUTING.md 존재
- 활발한 이슈/PR 활동
- 친절한 문서화 (초보자 친화적)
- 많은 contributor

### 1.7 Mistral 타입 (The Efficient Minimalist)
**특징**
- 미니멀리즘
- 효율성 극대화
- 간결한 코드
- 핵심에 집중

**판별 기준**
- 작은 코드베이스
- 낮은 의존성
- 명확한 단일 목적
- 간결한 README
- 높은 코드 밀도

### 1.8 Cohere 타입 (The Specialist)
**특징**
- 특정 도메인 전문화
- 깊이 있는 구현
- 성능 최적화
- 기술적 심화

**판별 기준**
- 특정 언어/프레임워크에 집중
- 성능 최적화 코드
- 벤치마크/프로파일링 포함
- 전문적인 문서
- 특정 분야의 깊은 커밋 히스토리

## 2. GitHub 분석 항목

### 2.1 커밋 분석
- **커밋 빈도**: 일평균 커밋 수, 주중/주말 패턴
- **커밋 크기**: 평균 변경 라인 수
- **커밋 메시지**: 길이, 구조화 정도, 규칙성
- **커밋 시간대**: 활동 시간 패턴

### 2.2 코드 복잡도
- **파일 크기**: 평균 파일 라인 수
- **함수 길이**: 평균 함수/메소드 길이
- **중첩 깊이**: 들여쓰기 레벨
- **의존성 수**: package.json, requirements.txt 등 분석

### 2.3 문서화 수준
- **README.md**: 존재 여부, 길이, 구조
- **주석 비율**: 코드 대비 주석 비율
- **문서 파일**: docs/, CONTRIBUTING.md, LICENSE 등
- **인라인 문서**: JSDoc, docstring 등

### 2.4 언어 다양성
- **사용 언어 수**: GitHub API의 languages
- **주 언어**: 가장 많이 사용된 언어
- **언어 분포**: 각 언어의 비중

### 2.5 프로젝트 완성도
- **README 완성도**: 설치, 사용법, 예제 포함 여부
- **테스트 존재**: test/, spec/ 디렉토리
- **CI/CD**: .github/workflows, .travis.yml 등
- **버전 관리**: 태그, 릴리즈 사용
- **활동성**: 최근 업데이트 시기

### 2.6 협업 활동
- **이슈 활동**: 생성, 해결 비율
- **PR 활동**: 작성, 리뷰 참여
- **Contributor 수**: 기여자 다양성
- **코드 리뷰**: PR의 리뷰 댓글 수
- **Discussion**: 토론 참여도

## 3. 사용자 플로우

### 3.1 홈 화면
```
[AI Personality Analyzer]
당신의 GitHub 코딩 성향을 AI 모델로 분석해보세요!

[GitHub Repository URL 입력]
예: https://github.com/username/repo

[분석하기 버튼]
```

### 3.2 URL 입력 및 검증
1. 사용자가 GitHub URL 입력
2. URL 형식 검증
   - github.com/[username]/[repo] 형식 확인
3. Repository 존재 여부 확인
   - GitHub API로 레포지토리 접근 가능한지 체크
4. Private 레포지토리 처리
   - Public 레포지토리만 지원 안내

### 3.3 분석 진행
```
[로딩 화면]
🔍 레포지토리 분석 중...
⚙️ 커밋 패턴 분석 중... (25%)
📊 코드 복잡도 계산 중... (50%)
📝 문서화 수준 평가 중... (75%)
✨ AI 타입 결정 중... (100%)
```

### 3.4 결과 표시
```
[결과 화면]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
당신은...

🎯 GPT-4 타입 (The Perfectionist)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[특징]
• 완벽주의적 접근
• 철저한 문서화와 주석
• 높은 코드 품질과 일관성

[분석 결과]
📊 커밋 패턴: ★★★★★ (매우 규칙적)
🔧 코드 복잡도: ★★★☆☆ (적절함)
📝 문서화: ★★★★★ (매우 상세함)
🌐 언어 다양성: ★★★☆☆ (보통)
✅ 완성도: ★★★★☆ (높음)

[당신과 비슷한 유명 레포지토리]
• facebook/react
• microsoft/typescript

[공유하기]
[트위터] [링크 복사]
```

### 3.5 공유 기능
- **트위터 공유**: 사전 작성된 트윗 템플릿
  ```
  나의 GitHub 코딩 성향은 "GPT-4 타입 (The Perfectionist)"!
  당신은 어떤 AI 모델 타입인가요?
  #AIPersonalityAnalyzer
  [URL]
  ```
- **링크 복사**: 결과 페이지 URL (쿼리 파라미터로 결과 저장)
- **이미지 다운로드**: 결과를 카드 형식 이미지로 저장

## 4. 테스트 계획

### 4.1 URL 검증 테스트
**TC-001: 올바른 GitHub URL**
- 입력: `https://github.com/facebook/react`
- 예상: 분석 진행

**TC-002: 잘못된 URL 형식**
- 입력: `invalid-url`, `http://google.com`
- 예상: 에러 메시지 "유효한 GitHub URL을 입력해주세요"

**TC-003: 존재하지 않는 레포지토리**
- 입력: `https://github.com/nonexistent/repo12345`
- 예상: 에러 메시지 "레포지토리를 찾을 수 없습니다"

**TC-004: Private 레포지토리**
- 입력: Private repo URL
- 예상: 에러 메시지 "Public 레포지토리만 분석 가능합니다"

### 4.2 분석 기능 테스트
**TC-101: 문서화가 잘 된 레포지토리**
- 입력: `facebook/react`, `microsoft/typescript`
- 예상: GPT-4 또는 Claude Opus 타입

**TC-102: 미니멀한 레포지토리**
- 입력: 작고 간결한 라이브러리
- 예상: Mistral 타입

**TC-103: 다양한 언어 사용 레포지토리**
- 입력: 여러 언어로 구성된 프로젝트
- 예상: Gemini 타입

**TC-104: 빈 레포지토리**
- 입력: 커밋이 없는 새 레포지토리
- 예상: 적절한 메시지 "분석할 데이터가 부족합니다"

### 4.3 성능 테스트
**TC-201: 대형 레포지토리**
- 입력: 10,000+ 커밋의 대형 프로젝트
- 예상: 30초 이내 분석 완료

**TC-202: 동시 요청**
- 시나리오: 10명이 동시에 분석 요청
- 예상: 모든 요청 정상 처리

### 4.4 UI/UX 테스트
**TC-301: 반응형 디자인**
- 테스트: 모바일, 태블릿, 데스크톱
- 예상: 모든 디바이스에서 정상 표시

**TC-302: 로딩 상태**
- 테스트: 분석 진행 중 UI
- 예상: 진행률 표시, 사용자에게 피드백

**TC-303: 에러 처리**
- 테스트: API 에러, 네트워크 에러
- 예상: 사용자 친화적 에러 메시지

### 4.5 통합 테스트
**TC-401: 전체 플로우**
1. URL 입력
2. 분석 진행
3. 결과 표시
4. 공유 버튼 클릭
- 예상: 모든 단계 정상 동작

**TC-402: 브라우저 호환성**
- 테스트: Chrome, Firefox, Safari, Edge
- 예상: 모든 브라우저에서 정상 동작

### 4.6 보안 테스트
**TC-501: XSS 방지**
- 입력: `<script>alert('xss')</script>` 포함 URL
- 예상: 스크립트 실행되지 않음

**TC-502: Rate Limiting**
- 시나리오: 짧은 시간에 많은 요청
- 예상: 적절한 제한 및 안내 메시지

## 7. 예상 제한사항 및 해결책

### 7.1 GitHub API Rate Limit
**문제**: 시간당 60회 (미인증) 또는 5000회 (인증) 제한
**해결책**:
- OAuth 인증 구현
- 결과 캐싱 (24시간)
- Rate limit 경고 표시

### 7.2 Private Repository
**문제**: 접근 불가
**해결책**: Public 레포지토리만 지원 명시

### 7.3 분석 정확도
**문제**: 새 레포지토리나 데이터 부족
**해결책**: 최소 요구사항 설정 (최소 10 커밋, 5 파일 등)
